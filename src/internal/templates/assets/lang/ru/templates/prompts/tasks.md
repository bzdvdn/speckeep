# Prompt задач SpecKeep

Вы создаете или обновляете `.speckeep/specs/<slug>/plan/tasks.md`.

## Goal

Разбейте согласованный план на исполнимые implementation tasks.

## Phase Contract

Inputs: `.speckeep/constitution.md`, `.speckeep/specs/<slug>/plan/plan.md`; опционально `.speckeep/specs/<slug>/summary.md`, `.speckeep/specs/<slug>/plan/data-model.md`, `.speckeep/specs/<slug>/plan/contracts/`, если декомпозиция их требует.
Outputs: `.speckeep/specs/<slug>/plan/tasks.md` с фазовым списком задач и секцией Acceptance Coverage.
Stop if: plan.md отсутствует, план слишком расплывчат, или хотя бы один AC нельзя привязать к выполнимой задаче.

## Режим работы

- Декомпозируйте только один согласованный plan package.
- Используйте `plan.md` как entrypoint и углубляйтесь только по необходимости.
- Стройте минимальный task list, который все еще безопасно покрывает фичу.
- Предпочитайте явную последовательность расплывчатым umbrella tasks.

## Load First

Всегда сначала прочитайте:

- `.speckeep/constitution.summary.md` если присутствует; иначе `.speckeep/constitution.md`
- `.speckeep/specs/<slug>/plan/plan.md`

## Load If Present

Читайте когда декомпозируемая задача явно ссылается на или зависит от содержимого этих файлов:

- `.speckeep/specs/<slug>/summary.md` (или `spec.md`) — когда `AC-*`, на который ссылается задача, имеет неоднозначный scope или границу приёмки
- `.speckeep/specs/<slug>/plan/data-model.md` — когда задачи должны создавать или менять сущности, поля, инварианты или переходы состояния
- `.speckeep/specs/<slug>/plan/contracts/` — когда задачи должны создавать или менять API endpoints, event payloads или integration boundaries
- `.speckeep/specs/<slug>/plan/research.md` — только если зафиксированный finding меняет порядок задач или вносит риск, влияющий на декомпозицию
- code files — только файлы, нужные для определения начала и конца implementation surfaces (напр., существующие сигнатуры функций, границы модулей)

Не считайте, что `research.md` обязан существовать; используйте его только когда план явно зависит от сохраненной неопределенности, внешней зависимости или зафиксированного trade-off.

## Do Not Read By Default

- implementation-файлы, не нужные для декомпозиции работы
- широкую историю репозитория

## Flags

`--repair <task-id-list>`: режим точечного исправления — исправьте конкретные задачи, выявленные verify или review, не переписывая весь task list.

Когда `--repair` присутствует в аргументах пользователя:
- Сначала прочитайте существующий `tasks.md`.
- Найдите только задачи, указанные в аргументе (например, `--repair T2.3,T3.1`).
- Для каждой названной задачи: обновите описание, outcome, `Touches:` или маппинг `AC-*` по необходимости.
- Не перестраивайте фазы, не перенумеровывайте другие задачи и не переписывайте секции, которые не затронуты ремонтом.
- Если repair выявляет проблему в самом плане, остановитесь и предложите `/speckeep.plan <slug> --update` вместо тихого расширения scope.
- Обновите `## Покрытие критериев приемки` только если repair-задачи меняют маппинг покрытия.
- Обновите `## Surface Map` только если изменились значения `Touches:`.

## Stop Conditions

Остановитесь и запросите refinement, если:

- `.speckeep/specs/<slug>/plan/plan.md` отсутствует
- задачи получаются расплывчатыми из-за недостаточно конкретного плана
- текущей декомпозиции не хватает spec, data model, contracts или research
- конституция блокирует предлагаемую декомпозицию
- декомпозиция расползается на несколько feature slug или несвязанные change sets
- один или несколько acceptance criteria нельзя сопоставить с исполнимой работой без догадок

Не переходите вперед к реализации.

## Инварианты

- Задачи ДОЛЖНЫ соответствовать плану и конституции.
- Используйте `plan.md` как entrypoint для декомпозиции.
- Не читайте нерелевантные артефакты других фич, чтобы компенсировать слабый план.
- Читайте implementation code только если без этого task list останется расплывчатым; предпочитайте узкий срез файлов, а не широкое исследование.
- Когда доступен `/.speckeep/scripts/check-tasks-ready.*`, предпочитайте запускать его как phase readiness check, а не читать исходники scripts.
- Не читайте `/.speckeep/scripts/*` по умолчанию, если только не отлаживаете сами скрипты, не работаете над самим SpecKeep или пользователь явно не просит разобрать script logic.
- Task list должен быть исполним по порядку.
- Каждый критерий приемки должен быть покрыт хотя бы одной задачей.
- Предпочитайте конкретные, проверяемые и implementation-oriented задачи.
- Добавляйте шаги валидации и синхронизации документации только когда они действительно нужны.
- Не создавайте расплывчатые umbrella tasks.
- Task list должен быть читаемым и для implementation-агента, и для reviewer без дополнительной интерпретации.
- Targeted code reading во время декомпозиции полезен, если он уменьшает повторное чтение на этапе implementation.
- Не начинайте implementation work, не редактируйте исходный код и не заявляйте, что задачи уже выполнены, на фазе tasks.

## Правила языка

- Используйте настроенный в проекте язык документации для нового или обновляемого содержимого задач.
- Уважайте уже сложившееся локальное соглашение task-документа только если без этого существующий файл станет непоследовательным.
- Не смешивайте языки внутри одного task list без сильной причины.
- Загружайте более глубокие артефакты только когда это действительно нужно текущей декомпозиции.

## Правила формата задач

- Следуйте структуре шаблона `.speckeep/templates/tasks.md`: группируйте задачи по упорядоченным фазам (`## Фаза N: Название`).
- Каждая задача ДОЛЖНА иметь phase-scoped task ID в формате `T<phase>.<index>`.
- Каждая задача ДОЛЖНА иметь формат: `- [ ] T<phase>.<index> <глагол действия> — <конкретный измеримый результат>`
- Каждая задача SHOULD по возможности ссылаться на 1-2 стабильных ID (`AC-*`, `RQ-*`, `DEC-*`).
- Каждая задача ДОЛЖНА содержать поле `Touches:` с конкретными файлами или модулями, которые задача создаёт или меняет. Это основной механизм предотвращения повторных чтений на фазе implement — агент использует их для batch-чтения всех нужных файлов за один проход. Держите компактно (`Touches: src/auth/handler.ts, src/session/store.ts`). Используйте module-level имена только когда точный файл ещё нельзя определить (`Touches: src/auth/`). Задача без `Touches:` вынуждает implement-агента на exploratory reads, что расходует токены.
- Секция `## Surface Map` ДОЛЖНА быть размещена перед первой фазой: двухколоночная таблица, сопоставляющая каждую уникальную implementation surface с task IDs (`Surface | Tasks`). Это манифест batch-чтения для implement-агента — без него агент вынужден сканировать каждую строку задачи для построения списка чтения.
- Задачи в совокупности ДОЛЖНЫ покрывать все acceptance criteria из спецификации. Если критерий не покрыт ни одной задачей — это blocker.
- Секция `## Покрытие критериев приемки` ДОЛЖНА содержать хотя бы одну явную строку покрытия для каждого критерия приемки.
- Строки покрытия SHOULD ссылаться на стабильные acceptance IDs и task IDs, например `AC-001 -> T1.1, T2.1`.
- Критерий приемки считается покрытым только когда ВСЕ задачи, на него ссылающиеся, завершены. Если хотя бы одна привязанная задача открыта — критерий ещё не выполнен; verify должен считать его незавершённым независимо от статуса остальных привязанных задач.
- Для новых task lists task IDs обязательны.
- Если существующий task list без task IDs существенно редактируется, нормализуйте его к ID-based формату.

## Правила качества содержимого

- У каждой фазы должна быть короткая цель, объясняющая, зачем эта фаза существует.
- **Ленивая декомпозиция**: Предпочитайте несколько конкретных задач (5-10 на фичу) с измеримым результатом множеству мелких bookkeeping items. Не создавайте «микро-задачи» (на 1-5 строк кода) на фазе tasks; агент реализации уточнит их по месту, если потребуется.
- Фокусируйтесь на «этапных» задачах, привязанных к конкретным файлам или функциональным границам.
- Держите outcome-часть каждой задачи в рамках ≤ 12 слов. Если нужно больше — задача недостаточно конкретна, разбейте её или уточните глагол.
- Если acceptance proof простой, встраивайте его прямо в outcome вместо обращения к spec: лучше `add POST /auth/login — возвращает 200 с полем JWT token — AC-001`, чем `add login handler — endpoint работает — AC-001`.
- Используйте action verbs, привязанные к наблюдаемой работе: implement, add, migrate, validate, remove, backfill, document.
- Отделяйте foundational setup от core behavior и отдельно выносите proof/validation вместо смешивания его с общей реализацией.
- Держите значения `Touches:` как конкретные file paths где возможно (`src/auth/handler.ts`), а не абстрактные концепции (`auth flow`). Implement-агенту нужны пути, а не описания.
- Если задача нужна только для доказательства поведения, говорите об этом явно, а не прячьте это внутри большой implementation-задачи.
- Если фаза не нужна этой фиче, пропустите её или явно укажите, что она осознанно не требуется, вместо generic filler tasks.
- Текст задачи должен делать ожидаемый результат очевидным для reviewer без постоянного возврата в plan.
- Negative examples: избегайте `misc`, `cleanup as needed`, `провести всё до конца`, `финальная полировка` и других формулировок, которые прячут результат за общим глаголом.

## Output expectations

- Запишите или patch-обновите `.speckeep/specs/<slug>/plan/tasks.md`
- Завершайте разговор summary block: `Slug`, `Status`, `Artifacts`, `Blockers`, `Готово к`
- Когда декомпозиция готова: `Готово к: /speckeep.implement <slug>`; если заблокировано — говорите прямо

## Self-Check

- У каждой задачи есть стабильный task ID и измеримый результат?
- Все acceptance criteria покрыты явно?
- Другой разработчик сможет выполнить эти задачи по порядку без догадок о том, что считается `done`?
