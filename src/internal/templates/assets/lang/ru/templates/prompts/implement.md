# Prompt реализации SpecKeep

Вы исполняете запланированную реализацию фичи.

## Goal

Реализуйте фичу, следуя существующему списку задач и не расширяя scope.

## Phase Contract

Inputs: `.speckeep/constitution.md`, `.speckeep/specs/<slug>/plan/tasks.md`; более глубокие артефакты — только если активная задача их требует.
Outputs: код реализации, обновлённые чекбоксы задач в `.speckeep/specs/<slug>/plan/tasks.md`.
Stop if: tasks.md отсутствует, следующая задача неконкретна, scope требует выдумывать новые задачи, или все задачи уже выполнены.

## Flags

`--continue`: режим возобновления — начните с первой незавершённой задачи, считая все ранее отмеченные задачи корректно выполненными.

Когда `--continue` присутствует в аргументах пользователя:
- Прочитайте `tasks.md` и пропустите все задачи, уже отмеченные `[x]`.
- Не перепроверяйте, не перечитывайте и не переделывайте завершённые задачи.
- Начните session-start batch read, используя только `Touches:` surfaces из оставшихся незавершённых задач.
- Если первая незавершённая задача зависит от outputs завершённой задачи, которые не видны в ожидаемых файлах, остановитесь и сообщите о несоответствии вместо тихого переделывания работы.
- Progress output должен начинаться с первого незавершённого task ID, а не с T1.1.

`--phase <номер>`: выполнять только указанную фазу.

`--tasks <список-task-id>`: выполнять только указанные task IDs.

Не принимайте `--phase` и `--tasks` одновременно в одном запуске.

## Режим работы

- Используйте `tasks.md` как execution entrypoint.
- Выполняйте минимальный безопасный scope, который разрешен запросом.
- Читайте только те артефакты и code files, которые нужны для активной задачи.
- По возможности patch-обновляйте существующие файлы вместо широкого переписывания.
- Предпочитайте readiness scripts, когда они доступны, перед чтением более глубоких артефактов.

## Load First

Всегда сначала прочитайте это перед началом реализации:

- `.speckeep/constitution.summary.md` если присутствует; иначе `.speckeep/constitution.md`
- `.speckeep/specs/<slug>/plan/tasks.md`

## Load If Present

Читайте когда активная задача явно ссылается на или зависит от содержимого этих файлов:

- `.speckeep/specs/<slug>/summary.md` (или `spec.md`) — когда задача ссылается на `AC-*`, чей intent или граница scope неоднозначны из текста задачи
- `.speckeep/specs/<slug>/plan/plan.md` — когда задача ссылается на `DEC-*`, sequencing constraint или implementation surface, не полностью описанные в задаче
- `.speckeep/specs/<slug>/plan/data-model.md` — когда задача создаёт или меняет сущности, поля, инварианты или переходы состояния
- `.speckeep/specs/<slug>/plan/contracts/` — когда задача создаёт или меняет API endpoints, event payloads или integration boundaries
- `.speckeep/specs/<slug>/plan/research.md` — только если задача зависит от зафиксированного trade-off или finding по внешней зависимости
- code files — только те, что перечислены в `Touches:` активных задач; начинайте с surfaces из plan/tasks перед расширением

Не считайте, что `research.md` обязан существовать; используйте его только когда активная задача зависит от сохраненной неопределенности, внешней зависимости или зафиксированного trade-off.

## Do Not Read By Default

- нерелевантные контракты
- весь код репозитория, если реально нужны только несколько файлов
- большую историческую переписку, если нет блокера

## Stop Conditions

Остановитесь и запросите refinement, если:

- `.speckeep/specs/<slug>/plan/tasks.md` отсутствует
- следующая задача недостаточно конкретна для безопасной реализации
- текущей задаче не хватает spec, plan, data model или contracts
- план конфликтует с конституцией
- реализация требует scope за пределами текущего task list
- выбранная работа заставляет менять другой feature package или slug, который не входит в текущий task scope
- следующий безопасный шаг потребовал бы придумывать новые задачи или acceptance coverage

Если все задачи в `tasks.md` уже отмечены выполненными — сообщите об этом и не продолжайте реализацию.

Не расширяйте scope, чтобы обойти эти проблемы.

## Правила scope

- Поведение по умолчанию: если пользователь явно не ограничил scope, выполняйте только первую незавершённую фазу. Фаза считается «незавершённой», если в ней есть хотя бы одна открытая задача. Завершите все открытые задачи текущей фазы, прежде чем переходить к следующей. Не пересекайте границу фазы, пока все задачи в текущей фазе не закрыты.
- В режиме `--continue` начинайте с первой незавершённой задачи независимо от границ фаз.
- В режиме `--phase` или `--tasks` сохраняйте порядок из `tasks.md`, а не придумывайте новый порядок из текста запроса.
- Если выбранная фаза или task IDs отсутствуют в `tasks.md`, остановитесь и запросите refinement.
- Если scoped execution перескочил бы через незавершённую более раннюю работу, остановитесь и перечислите, какие незавершённые задачи будут пропущены. Не продолжайте, пока пользователь явно не подтвердит, что хочет их пропустить.

## Дисциплина чтения

**Каждый файл читается один раз за сессию.** Перечитывайте только если файл изменён извне или задача требует проверить состояние после неочевидного изменения предыдущей задачей.

**Начало сессии** (перед первой задачей):
1. Запустите `.speckeep/scripts/check-implement-ready.*`, если доступен.
2. Прочитайте `tasks.md`; используйте `## Surface Map` как манифест batch-чтения — он перечисляет все implementation surfaces и какие задачи их затрагивают. Если Surface Map отсутствует, соберите `Touches:` fields из задач в scope.
3. Прочитайте все surfaces из манифеста за один проход. Выполняйте задачи из предзагруженного контекста — не перечитывайте файлы между задачами.

## Инварианты

- Реализуйте только незавершенные задачи из `tasks.md`.
- Соблюдайте порядок и фазовую структуру в `tasks.md`.
- Не перепроектируйте и не перепланируйте фичу молча во время реализации.
- Если вы изменили файл, не указанный в `Touches:` активной задачи, остановитесь и объясните почему до того, как продолжить. Незаявленные изменения поверхностей — это нарушение scope, а не улучшение.
- Не читайте по умолчанию нерелевантные артефакты других фич или широкие области репозитория.
- Не перечитывайте файлы между задачами; опирайтесь на batch-чтение из начала сессии и историю собственных правок.
- Когда доступен `/.speckeep/scripts/check-implement-ready.*`, предпочитайте запускать его как phase readiness check, а не читать исходники scripts.
- Не читайте `/.speckeep/scripts/*` по умолчанию, если только не отлаживаете сами скрипты, не работаете над самим SpecKeep или пользователь явно не просит разобрать script logic.
- Если задачу нельзя безопасно реализовать из текущих артефактов, остановитесь и запросите refinement.
- Если для продолжения нужно сделать неочевидное допущение (форма API, формат данных, обработка ошибок), зафиксируйте его как `[ASSUMPTION: ...]` в progress output до того, как действовать. Если допущение существенно влияет на acceptance scope — остановитесь и спросите.
- Отмечайте завершенные задачи в `tasks.md`.
- **In-place Декомпозиция**: Если задача `T1.1` слишком сложна для отслеживания как единого целого, вы можете уточнить её, добавив вложенные подзадачи (напр., `- [ ] T1.1.1 Подзадача`).
- **Ограничения подзадач**:
  - Подзадачи НЕ ДОЛЖНЫ добавлять новые файлы в список `Touches:` родительской задачи.
  - Подзадачи НЕ ДОЛЖНЫ менять или расширять привязку `AC-*` родительской задачи.
  - Несколько подзадач МОГУТ затрагивать один и тот же файл из `Touches:` родительской задачи, но каждая подзадача обязана указать в тексте outcome свой конкретный вклад в этот файл. Подзадачи с общим файлом должны быть упорядочены так, чтобы последняя из них подтверждала финальное состояние файла.
  - Если декомпозиция выявляет, что исходный `plan.md` ошибочен или требует новой implementation surface, вы ОБЯЗАНЫ остановиться и запросить обновление плана или `repair`.
- **Аннотируйте код**: Каждое нетривиальное изменение должно включать комментарий со ссылкой на ID задачи и основной критерий приемки (AC), который оно удовлетворяет.
  Формат: `// @sk-task <TASK_ID>: <Краткое описание> (<AC_ID>)`
  Пример: `// @sk-task T1.1: Добавить метод экспорта в CSV (AC-001)`
  Для тестов (unit/integration/e2e) добавляйте отдельную аннотацию:
  Формат: `// @sk-test <TASK_ID>: <TestName> (<AC_ID>)`
  Пример: `// @sk-test T1.1: TestCSVExport (AC-001)`
  Если задача была декомпозирована на подзадачи, используйте в аннотации ID подзадачи (например, `T1.1.1`), а не родительский ID. Это сохраняет точность трассируемости при verify.
- Держите runtime updates короткими и привязанными к текущей фазе и task IDs.
- Не нарушайте конституцию.
- Оставляйте фичу в таком состоянии, чтобы следующая verify-проверка могла понять, что изменилось, что осталось и почему задача считается завершенной, без догадок.
- Не перепланируйте фичу, не выдавайте verify verdict и не закрывайте молча соседние задачи, которые не входили в активный execution scope.

## Правила прогресса

- Всегда явно сообщайте, какая фаза сейчас находится в работе, если активная реализация пересекает границу фаз.
- Когда фаза становится завершенной внутри текущего execution scope, выдавайте короткий phase-completion update с названием фазы и завершенными task IDs.
- Такие runtime progress updates должны быть на настроенном в проекте языке общения с агентом, чтобы пользователь не получал полностью английские phase-status сообщения в неанглийском workflow.
- Используйте короткие progress lines в стабильном формате:
  - `[T1.1] started`
  - `[T1.1] done`
  - `[T1.1] blocked: <reason>`
  - `[Phase 1] done: T1.1, T1.2`
- Загружайте более глубокие артефакты только когда этого требует текущая задача.

## Правила handoff

- Перед тем как отмечать задачу выполненной, убедитесь, что наблюдаемый результат из текста задачи действительно достигнут. Явно укажите: какой файл изменился, что добавлено или изменено и в чём состоит observable proof. Не отмечайте done, если единственное доказательство — «код выглядит правильно».
- Если задача ссылается на `AC-*`, держите реализацию в рамках этой acceptance scope и не расширяйте поведение молча.
- Когда активный scope завершен, оставляйте достаточно evidence для следующей фазы:
  - обновленные чекбоксы в `tasks.md`
  - краткое summary того, что изменилось
  - ясные blockers или оставшиеся открытые задачи
- Если по ходу реализации выясняется, что task text, acceptance coverage или план неверны, остановитесь и верните workflow на refinement `tasks` или `plan`, а не чините process contract молча в коде.

## Правила языка

- Следуйте предпочтительному языку комментариев в коде, который зафиксирован в `.speckeep/speckeep.yaml` и `.speckeep/constitution.md`.
- Когда добавляете или редактируете комментарии в коде, держите их на настроенном языке комментариев, если только в самом файле уже нет устойчивого локального соглашения, которое нужно сохранить.
- Не вводите смешанные по языку комментарии в одной локальной области кода без сильной причины.
- Если плана или задач недостаточно, остановитесь и запросите refinement вместо того, чтобы выдумывать широкий новый scope.

## Output expectations

- Выполните работу; обновите чекбоксы в `tasks.md`; сообщайте phase progress в формате `[T1.1] done`
- Явно укажите, какие acceptance criteria теперь покрыты; не заявляйте покрытие, которое не реализовано
- Завершайте разговор summary block: `Slug`, `Status`, `Artifacts`, `Blockers`, `Готово к`
- Следующая фаза после `implement` — всегда `verify`. Не предлагайте `/speckeep.archive` напрямую (архивирование допустимо только после verifу verdict `pass`).
- Когда scope готов к проверке: `Готово к: /speckeep.verify <slug>`

## Self-Check

- Я выполнил только тот scope, который разрешен `tasks.md` и запросом?
- Я обновил выполненные задачи и coverage по acceptance criteria?
- Фаза `verify` поймет, что изменилось и что осталось, без повторного чтения всего репозитория?
