# Prompt реализации SpecKeep

Исполняете запланированную реализацию фичи.

Следуйте базовым правилам в `AGENTS.md` (пути, git, load discipline, readiness scripts, язык, phase discipline).

## Цель

Реализовать фичу по существующему task list без расширения scope.

## Phase Contract

Inputs: `.speckeep/constitution.md`, `.speckeep/specs/<slug>/plan/tasks.md`; глубже — только если активная задача требует.
Outputs: код реализации, обновлённые чекбоксы в `tasks.md`.
Stop if: `tasks.md` нет, следующая задача неконкретна, scope требует выдумывать новые задачи, или все задачи выполнены.

## Flags

`--continue`: возобновление — с первой незавершённой задачи, считая предыдущие `[x]` корректными.

- Прочитайте `tasks.md`; пропустите `[x]`. Не перепроверяйте, не перечитывайте и не переделывайте.
- Session-start batch read — только `Touches:` остальных незавершённых задач.
- Если незавершённая задача зависит от outputs завершённой, но их нет в ожидаемых файлах — остановитесь и сообщите; не переделывайте молча.
- Progress output с первого незавершённого task ID, не с `T1.1`.

`--phase <номер>`: выполнять только указанную фазу.

`--tasks <список-task-id>`: выполнять только указанные task IDs.

Не принимайте `--phase` и `--tasks` одновременно.

## Режим работы

- `tasks.md` — execution entrypoint.
- Минимальный безопасный scope.
- Читайте только артефакты и код, нужные для активной задачи.
- Предпочитайте patch существующих файлов широкому переписыванию.
- Предпочитайте readiness scripts чтению более глубоких артефактов.

## Load First

Всегда до реализации:

- `.speckeep/constitution.summary.md` если есть; иначе `.speckeep/constitution.md`
- `.speckeep/specs/<slug>/plan/tasks.md`

## Load If Present

Только когда активная задача на них ссылается / от них зависит:

- `summary.md` (или `spec.md`) — задача ссылается на `AC-*`, неоднозначный из текста
- `plan/plan.md` — задача ссылается на `DEC-*`, sequencing constraint или surface, не описанные в задаче
- `plan/data-model.md` — задача создаёт/меняет сущности, поля, инварианты, transitions
- `plan/contracts/` — задача создаёт/меняет API endpoints, event payloads, integration boundaries
- `plan/research.md` — только если задача зависит от зафиксированного trade-off или внешней зависимости
- code files — только из `Touches:` активных задач; начинайте с plan/tasks surfaces

Не считайте, что `research.md` обязан существовать; используйте его только когда активная задача зависит от сохранённой неопределённости, внешней зависимости или зафиксированного trade-off.

## Do Not Read By Default

- нерелевантные contracts
- весь код репо, если нужны несколько файлов
- большая историческая переписка без блокера

## Stop Conditions

Остановитесь и запросите refinement, если:

- `tasks.md` отсутствует
- следующая задача неконкретна для безопасной реализации
- задаче не хватает spec/plan/data-model/contracts
- план конфликтует с конституцией
- реализация требует scope за пределами task list
- выбранная работа заставляет менять другой feature package или slug, который не входит в текущий task scope
- следующий безопасный шаг потребовал бы придумывать новые задачи или acceptance coverage

Все задачи `[x]` → сообщите и остановитесь. Не расширяйте scope, чтобы обойти проблемы.

## Правила scope

- Default behavior: if the user does not restrict scope, execute only the first unfinished phase. Фаза «незавершённая» = есть ≥ 1 открытая задача. Закройте все открытые в текущей фазе, прежде чем переходить. Не пересекайте границу фазы, пока не закрыты все задачи.
- `--continue`: с первой незавершённой задачи независимо от границ фаз.
- `--phase` / `--tasks`: сохраняйте порядок `tasks.md`, не придумывайте новый.
- Выбранные фаза/IDs отсутствуют в `tasks.md` → остановитесь и запросите refinement.
- Scoped execution пропустил бы раннюю незавершённую работу → остановитесь, перечислите пропуски, ждите явного подтверждения.

## Дисциплина чтения

**Каждый файл читается один раз за сессию.** Перечитывайте, только если файл изменён извне или задача требует проверки состояния после неочевидного изменения.

**Начало сессии** (перед первой задачей):

1. Запустите `.speckeep/scripts/check-implement-ready.*` если доступен (slug первый аргумент: `bash ./.speckeep/scripts/check-implement-ready.sh <slug>`).
2. Прочитайте `tasks.md`; `## Surface Map` — манифест batch-чтения. Нет — соберите `Touches:` из задач в scope.
3. Прочитайте все surfaces одним проходом. Выполняйте задачи из предзагруженного контекста — не открывайте файлы повторно.

## Инварианты

- Реализуйте только незавершённые задачи. Соблюдайте порядок и фазовую структуру.
- Никогда не перепланируйте фичу молча.
- Изменение файла вне `Touches:` — нарушение scope: остановитесь и объясните.
- Не читайте нерелевантные артефакты или области репо по умолчанию. Не перечитывайте файлы между задачами.
- Если `/.speckeep/scripts/check-implement-ready.*` доступен — запускайте (slug первый аргумент: `bash ./.speckeep/scripts/check-implement-ready.sh <slug>` или PowerShell `.\.speckeep\scripts\check-implement-ready.ps1 <slug>`) вместо чтения исходников.
- Задачу нельзя безопасно реализовать из текущих артефактов → стоп и refinement.
- Неочевидное допущение (форма API, формат данных, error handling) → `[ASSUMPTION: ...]` в progress output до действий. Существенно влияет на acceptance scope → стоп и спросите.
- Отмечайте завершённые задачи в `tasks.md`.
- **In-place декомпозиция**: `T1.1` слишком сложна → индентованные подзадачи (`- [ ] T1.1.1 Подзадача`).
- **Ограничения подзадач**:
  - НЕ добавляйте файлы в `Touches:` родителя.
  - НЕ меняйте привязку `AC-*` родителя.
  - Несколько подзадач могут трогать один файл из `Touches:`; каждая указывает свой вклад, последняя по порядку подтверждает финальное состояние.
  - Декомпозиция показала, что `plan.md` ошибочен или нужна новая surface → стоп и request plan update или `repair`.
- **Аннотации кода** для каждого нетривиального изменения:
  - `// @sk-task <TASK_ID>: <Краткое описание> (<AC_ID>)` — пример: `// @sk-task T1.1: Добавить метод экспорта в CSV (AC-001)`
  - Для тестов: `// @sk-test <TASK_ID>: <TestName> (<AC_ID>)` — пример: `// @sk-test T1.1: TestCSVExport (AC-001)`
  - Декомпозирована → используйте ID подзадачи (`T1.1.1`), не родительский.
- Держите runtime updates короткими и привязанными к текущей фазе и task IDs.
- Не нарушайте конституцию.
- Оставьте фичу в состоянии, которое следующий verify может инспектировать без догадок о том, что изменилось, что осталось и почему задача завершена.
- Не перепланируйте фичу, не выдавайте verify verdict и не закрывайте молча соседние задачи вне активного scope.

## Правила прогресса

- Явно сообщайте активную фазу при пересечении границ.
- Завершение фазы в активном scope → короткий phase-completion update с именем фазы и task IDs.
- Runtime progress updates — на настроенном языке (не английский в неанглийском workflow).
- Короткие progress lines:
  - `[T1.1] started`
  - `[T1.1] done`
  - `[T1.1] blocked: <reason>`
  - `[Phase 1] done: T1.1, T1.2`
- Загружайте глубокие артефакты только когда текущая задача требует.

## Правила handoff

- Перед `[x]` подтвердите, что observable outcome из текста задачи реально есть. Укажите: какой файл изменён, что добавлено/изменено, какой observable proof. Не отмечайте, если единственное доказательство — «код выглядит правильно».
- Задача ссылается на `AC-*` → не расширяйте поведение молча.
- Финал активного scope → оставьте evidence: чекбоксы, краткое summary изменений, явные blockers или оставшиеся задачи.
- Реализация показала, что task text / acceptance coverage / план неверны → стоп и вернитесь в `tasks`/`plan` refinement; не чините process contract молча в коде.

## Output

- Реализуйте работу; обновите чекбоксы; `[T1.1] done`-lines.
- Укажите покрытые AC; не заявляйте непокрытое покрытие.
- Завершайте summary block: `Slug`, `Status`, `Artifacts`, `Blockers`, `Готово к`.
- Следующая фаза после `implement` — всегда `verify`. Не предлагайте `/speckeep.archive` напрямую.
- Готово: `Готово к: /speckeep.verify <slug>`.
