# Prompt плана SpecKeep

Вы создаете или обновляете implementation plan package для одной фичи.

## Goal

Сформируйте технические planning artifacts для спецификации в `<specs_dir>/<slug>/plan/` (по умолчанию: `.speckeep/specs/<slug>/plan/`).

## Примечание о путях

Пути в этом промпте показаны для layout по умолчанию. Если в `.speckeep/speckeep.yaml` переопределены `paths.specs_dir` или `project.constitution_file`, всегда следуйте путям из конфигурации, а не примерам по умолчанию.
Читайте `.speckeep/speckeep.yaml` максимум один раз за сессию для резолва путей; не перечитывайте его без необходимости (только если конфиг изменился или путь неоднозначен).

## Phase Contract

Inputs: `.speckeep/constitution.md`, `.speckeep/specs/<slug>/spec.md`, `.speckeep/specs/<slug>/inspect.md`, узкий контекст репозитория.
Outputs: `.speckeep/specs/<slug>/plan/plan.md`, `.speckeep/specs/<slug>/plan/data-model.md`; опционально `.speckeep/specs/<slug>/plan/contracts/`, `.speckeep/specs/<slug>/plan/research.md`.
Stop if: spec или inspect отсутствуют, spec слишком расплывчата для архитектурных решений, или конфликт с конституцией.

## Режим работы

- Планируйте только одну фичу.
- Предпочитайте patch существующих артефактов вместо широкого переписывания.
- Держите контекст узким и привязанным к реальности репозитория.
- Создавайте только те артефакты, которые действительно оправданы фичей.
- **Не создавайте и не переключайте ветки.** Feature branch должна уже существовать с фазы spec. Если вы не находитесь на ожидаемой feature branch — остановитесь и сообщите об этом, не создавайте её.
- **Не создавайте `tasks.md`.** Декомпозиция на задачи — отдельная фаза, запускаемая командой `/speckeep.tasks`. Остановитесь после записи план-артефактов и выведите строку next command.

## Флаги

`--update`: режим точечного обновления — обновить конкретную секцию, решение (`DEC-*`), implementation surface или добавить контракт без переписывания всего plan package. Аналогично `spec --amend`.

Когда `--update` присутствует в аргументах пользователя:
- Сначала прочитайте существующие plan artifacts.
- Примените только то изменение, которое описано в оставшихся аргументах.
- Не реструктуризируйте и не переписывайте секции, которые не затрагиваются.
- Если обновление меняет `DEC-*`, обновите поля `Affects` и `Validation`, если они стали неактуальны.
- Если обновление меняет `data-model.md` или `contracts/`, обеспечьте согласованность со ссылками в `plan.md`.
- Не инвалидируйте downstream `tasks.md`, если изменение не затрагивает материально декомпозицию задач или acceptance coverage.

`--research`: сначала выполнить исследование, прежде чем создавать план.

Если `--research` присутствует в аргументах пользователя:
- Прочитайте spec и inspect report, затем определите 1–5 конкретных unknowns, которые на данный момент блокируют планирование.
- Запишите их в `.speckeep/specs/<slug>/plan/research.md`.
- Остановитесь после записи `research.md` и спросите: «Исследование завершено — перейти к плану?»
- Ждите явного подтверждения перед созданием `plan.md`, `data-model.md` или любых других planning artifacts.
- Если пользователь подтверждает, продолжайте обычный planning flow, используя findings из research как основу.
- Если пользователь говорит остановиться, завершайте сессию, где `research.md` — единственный новый артефакт.

Не создавайте `plan.md` в том же проходе, что и `--research`, если только пользователь явно не подтверждает это.

Не принимайте `--update` и `--research` одновременно в одном запуске. Если оба флага присутствуют, остановитесь и спросите пользователя, какой режим использовать.

## Load First

- `.speckeep/constitution.md`
- `.speckeep/specs/<slug>/spec.md`
- `.speckeep/specs/<slug>/inspect.md`
- только те файлы кода и документации репозитория, которые нужны для планирования этой одной фичи
- если нужно читать код, предпочитайте минимальный набор файлов, достаточный для определения concrete implementation surfaces, границ и ограничений

## Do Not Read By Default

- большие области репозитория, не влияющие на эту фичу
- optional `research.md`, если неопределенность еще не подтверждена

## Stop Conditions

Остановитесь и запросите уточнение или refinement, если:

- `.speckeep/specs/<slug>/spec.md` не существует
- `.speckeep/specs/<slug>/inspect.md` не существует
- спецификация слишком расплывчата для архитектурных решений, contracts или data model
- ограничения конституции конфликтуют с предполагаемым планом
- план пришлось бы вести через неясную integration- или architectural-boundary, которую не подтверждают spec и сфокусированный контекст репозитория
- работа имеет смысл только если одновременно планировать несколько feature packages

Не компенсируйте это чтением широкого нерелевантного контекста репозитория.

Если отсутствует `spec.md` или `inspect.md`, не пытайтесь создавать их во время фазы `plan`. Остановитесь и попросите пользователя запустить:

- `Готово к: /speckeep.spec <slug>`
- затем `Готово к: /speckeep.inspect <slug>`
- затем повторить `Готово к: /speckeep.plan <slug>`

## Required outputs

Всегда создавайте или обновляйте:

- `.speckeep/specs/<slug>/plan/plan.md`
- `.speckeep/specs/<slug>/plan/data-model.md` — создавайте когда фича вводит или изменяет persisted state, форму сущностей или state transitions; иначе создайте файл с единственной строкой: «Эта фича не вводит новых сущностей.»

Создавайте только если фича действительно затрагивает:

- `.speckeep/specs/<slug>/plan/contracts/api.md` — только если фича затрагивает API boundaries
- `.speckeep/specs/<slug>/plan/contracts/events.md` — только если фича использует или публикует события

Создавайте `.speckeep/specs/<slug>/plan/research.md` только если выполняется хотя бы одно из условий:

- фича зависит от внешней системы, API или зависимости, чье поведение пока неясно
- есть несколько реалистичных implementation options с существенными trade-off'ами, которые нужно сохранить
- есть неочевидный performance, security, reliability или integration risk, влияющий на планирование
- перед конкретизацией плана нужно исследовать repository constraint или архитектурную границу

Перед созданием `research.md` сначала зафиксируйте конкретные unknowns:

- перечислите только 1-5 конкретных неизвестных, которые реально блокируют планирование
- привязывайте каждый unknown к решению, риску или границе именно этой фичи
- не исследуйте технологию или подсистему “вообще”; исследуйте только узкий вопрос, который меняет план
- если конкретных unknowns больше не осталось, `research.md` создавать не нужно

## Инварианты

- План ДОЛЖЕН соответствовать конституции.
- Inspect — обязательный prerequisite для planning. Не планируйте по spec, которая не прошла inspect и не имеет сохраненного отчета.
- Держите planning привязанным к текущей спецификации, а не к идеализированной архитектуре.
- Не читайте нерелевантные артефакты других фич, чтобы компенсировать нехватку ясности.
- Читайте код узко: ровно настолько, чтобы зафиксировать implementation surfaces, integration boundaries и repository constraints для этой фичи.
- Когда доступен `/.speckeep/scripts/check-plan-ready.*`, предпочитайте запускать его как phase readiness check, а не читать исходники scripts.
- Важно: readiness wrapper запускается со slug как первым аргументом. Пример: `bash ./.speckeep/scripts/check-plan-ready.sh <slug>` (или PowerShell: `.\.speckeep\scripts\check-plan-ready.ps1 <slug>`).
- Не читайте `/.speckeep/scripts/*` по умолчанию, если только не отлаживаете сами скрипты, не работаете над самим SpecKeep или пользователь явно не просит разобрать script logic.
- Предпочитайте конкретные implementation decisions, а не общие советы.
- Фиксируйте технологии, библиотеки, framework choices и version constraints только когда они реально влияют на implementation shape, integration boundaries, validation или risk.
- Если называете версию или зависимость, объясняйте, почему это важно для этой фичи: compatibility, repository constraint, внешний contract, rollout risk или validation.
- Не перечисляйте stack details ради полноты; фиксируйте только те technical constraints, которые уменьшают downstream guesswork.
- Опциональные артефакты должны оставаться опциональными; не создавайте их по привычке.
- Не создавайте `research.md` ради общего brainstorming или очевидной реализации, которую уже можно спланировать из spec и repository evidence.
- План считается достаточным только тогда, когда downstream tasks можно декомпозировать без догадок.
- План ДОЛЖЕН явно называть concrete implementation surfaces, которые предполагается изменить.
- План ДОЛЖЕН сопоставлять каждый acceptance criterion с подходом к реализации до того, как будут писаться `tasks`.
- Не записывайте во время planning сам task checklist, не редактируйте implementation code и не выдавайте verify/archive conclusions.
- Если автору следующей фазы пришлось бы гадать о методе, границе или пути валидации для `AC-*`, план недостаточно конкретен.
- План должен быть достаточно конкретным, чтобы и агент, и reviewer видели форму реализации, tradeoff'ы и rollout implications без повторного чтения всего репозитория.
- Targeted code reading на этапе planning приветствуется, если он уменьшает downstream guesswork; широкое исследование репозитория не допускается.

## Правила языка

- Используйте настроенный в проекте язык документации для всех новых или обновляемых planning artifacts.
- Держите язык `plan.md`, `data-model.md`, `contracts/` и optional `research.md` внутренне согласованным.
- Уважайте уже сложившееся локальное соглашение документа только если без этого сохранение существующего артефакта станет непоследовательным.

## Правила трассируемости

- Следуйте структуре шаблонов `.speckeep/templates/plan.md` и `.speckeep/templates/data-model.md` при создании новых файлов.
- Data model и contracts ДОЛЖНЫ быть согласованы со спецификацией и её acceptance criteria.
- Ссылайтесь на стабильные acceptance IDs из спецификации, когда описываете acceptance-critical behavior.
- Если план фиксирует значимое implementation decision, записывайте его через стабильный decision ID, например `DEC-001`.
- Каждое значимое `DEC-*` SHOULD включать `Why`, `Affects` и `Validation`.
- Если фича вводит или меняет persisted state, переходы состояния или lifecycle rules, фиксируйте это в `data-model.md`.
- Если фича проходит через API boundary, фиксируйте request, response и error behavior в `contracts/api.md`.
- Если фича публикует или потребляет события, фиксируйте producer, consumer, payload и delivery assumptions в `contracts/events.md`.
- Не оставляйте форму сущностей, boundary IO или event payload только в prose внутри `plan.md`.
- Каждая сущность или contract entry SHOULD ссылаться на оправдывающий её `AC-*`.
- Если `data-model.md` содержит реальные сущности (не только placeholder-строку), план ДОЛЖЕН содержать в `plan.md` однострочное обоснование с именами конкретных сущностей, инвариантов или lifecycle concerns, которые это требуют.
- Если создается `contracts/`, план ДОЛЖЕН содержать в `plan.md` однострочное обоснование с именем конкретной API- или event boundary, которая это требует. Опустите `contracts/` и явно укажите «Эта фича не вводит API или event boundaries», если они не применимы.
- Если более богатые артефакты не нужны, предпочитайте не создавать их.
- Опирайтесь на реальность репозитория, а не на идеализированную архитектуру.
- Если критической информации не хватает, задавайте только минимально необходимый уточняющий вопрос.
- Формулировки вроде `обновить backend по необходимости`, `подправить логику`, `провести через систему` считайте сигналом к refinement, а не завершённым планом.

## Правила качества содержимого

- `## Цель` должна описывать форму реализации, а не пересказывать пользовательский смысл из spec.
- `## Implementation Surfaces` должны объяснять, почему каждая surface меняется и новая она или существующая.
- `## Acceptance Approach` должен сопоставлять каждый `AC-*` с затрагиваемыми surfaces и observable proof.
- `## Данные и контракты` должны явно говорить, что меняется, что остается без изменений и почему.
- Каждое значимое `DEC-*` должно содержать `Why`, `Tradeoff`, `Affects` и `Validation`.
- `## Порядок реализации` должен отделять то, что обязано произойти сначала, от того, что можно распараллелить.
- `## Риски` должны включать mitigation, а не только название риска.
- `## Rollout и compatibility` должны быть явными, если важны migration, flags, compatibility или operational follow-up, и так же явно говорить, когда они не нужны.
- `## Проверка` должна быть привязана к `AC-*` или `DEC-*`, а не быть общим списком тестовых идей.
- Перед финализацией плана делайте короткий `Unknowns First` pass: если решение, surface или validation path все еще неясны, зафиксируйте unknown явно или остановитесь на refinement.
- Предпочитайте конкретные implementation-guidance формулировки архитектурному эссе. Если абзац не уменьшает downstream guesswork, его нужно ужать.
- `## Соответствие конституции` ДОЛЖНА присутствовать и ДОЛЖНА перечислять конкретные конституционные ограничения, применимые к этой фиче (например, «Необходимо использовать PostgreSQL согласно [CONST-DB]»), а затем подтверждать выполнение каждого из них или объяснять, как конфликт разрешается или откладывается. Голое «нет конфликтов» без перечисления проверенных ограничений недостаточно — это делает проверку соответствия неверифицируемой на фазе inspect.

## Output expectations

- Запишите или patch-обновите артефакты плана; укажите, какие опциональные артефакты созданы и почему
- Кратко опишите ключевые технические решения и явно укажите риски, блокирующие следующие фазы
- Завершайте разговор summary block: `Slug`, `Status`, `Artifacts`, `Blockers`, `Готово к`
- Когда планирование завершено: `Готово к: /speckeep.tasks <slug>`; если заблокировано — говорите прямо

## Self-Check

- Все опциональные артефакты действительно оправданы?
- Из этого plan package можно писать tasks без догадок?
- Reviewer быстро поймет форму реализации, основные риски и rollout story?
