# Prompt спецификации SpecKeep

Создаёте или обновляете одну feature specification.

## Goal

Сформируйте spec в `<specs_dir>/<slug>/spec.md` (по умолчанию: `.speckeep/specs/<slug>/spec.md`), соответствующую конституции.

## Phase Contract

Inputs: `.speckeep/constitution.md`, запрос пользователя, минимальный контекст репозитория.
Outputs: `.speckeep/specs/<slug>/spec.md` (создаётся или обновляется).
Stop if: цель неоднозначна, в запросе несколько фич, или AC пришлось бы выдумывать.

Если `.speckeep/speckeep.yaml` переопределяет `paths.specs_dir` или `project.constitution_file` — следуйте конфигу. Читайте его ≤1 раза за сессию.

Перед записью любого файла: создайте или переключитесь на `feature/<slug>` (или явный `--branch`) через `git switch [-c] <branch>`. Если переключение невозможно (нет git, detached HEAD) — остановитесь и сообщите причину.

## Флаги

`--name <название>`, опциональный `--slug`, опциональный `--branch`. Prompt-файл может содержать `name:`/`slug:`, но CLI-флаги приоритетнее. Slug выводится из `--name`/`name:`, если не задан явно. `--branch` переопределяет только ветку, не slug. Basename prompt-файла используется только когда он достаточно конкретен.

Если передан только `--name` без описания — запросите его и держите staged mode активным для следующего сообщения. Сообщение, начинающееся с `/speckeep.`, отменяет staged mode; любое другое — продолжение запроса.

`--amend`: точечная правка одной секции или одного `AC-*`.

- Читайте существующий spec; меняйте только запрошенное, нетронутые секции не переписывайте.
- Новый/изменённый AC ДОЛЖЕН быть в Given/When/Then со стабильным `AC-*` ID.
- Если правка материально меняет AC, scope или цель — inspect-отчёт инвалидирован: удалите `inspect.md` и `summary.md`, завершите `Готово к: /speckeep.inspect <slug>`.

`--greenfield`: режим greenfield-first.

- Используйте, когда репозиторий пустой, почти пустой или фича задаёт один из первых поставляемых продуктовых срезов.
- В этом режиме приоритетнее ясность user stories и MVP slice, чем детализация repository surfaces, которых ещё нет.
- Добавляйте `## User Stories`, `## MVP Slice` и `## First Deployable Outcome`, когда они уменьшают неопределённость следующей фазы.

## Load and Scope

Загружайте: `.speckeep/constitution.md`, запрос пользователя, минимум repo-контекста для снятия неоднозначности.
Не читайте по умолчанию: тяжёлый implementation-код, contracts/data-models других фич.
Если доступен `/.speckeep/scripts/check-spec-ready.*` — запускайте как readiness check, передавайте slug.

## Stop Conditions

Остановитесь и задайте минимальный вопрос, если:

- цель неоднозначна или запрос смешивает несколько фич
- чтобы удержать scope, нужны несколько feature slug или независимых спецификаций
- вход — prompt-файл с generic-именем без `name:`/`slug:`, и пользователь не передал `--name`/`--slug`
- вход больше похож на URL, чем на имя фичи
- AC пришлось бы выдумывать
- фича конфликтует с конституцией
- обязательная секция остаётся TBD/placeholder

Нужно уточнение? Делайте структурированный clarify-pass (1–3 точечных вопроса по пробелам, которые иначе заставят выдумывать AC или размоют scope), а не широкий интервью. В режиме `--greenfield` сначала тратьте этот бюджет на MVP slice, first deployable outcome и неоднозначности основного user flow. Как только ответов хватает — patch сразу.

Spec уже существует и актуален — сообщите и не меняйте.

## Инварианты

- Соответствие конституции. Одна фича. Patch > переписывание.
- Не записывайте в spec planning-решения, декомпозицию задач, implementation steps.
- AC выводятся из запроса и репо-реальности. Каждый AC — Given/When/Then со стабильным `AC-*`.
- `## Вне scope` и `## Допущения` обязательны. `## Критерии успеха` (`SC-*`) — только для измеримых целей (latency, throughput, error rate).
- Неясную деталь помечайте inline: `[NEEDS CLARIFICATION: что неизвестно и почему]` — inspect закроет до planning.
- Не фиксируйте технологии/библиотеки/версии по умолчанию. Упоминайте только когда это требование пользователя, существующий repo-constraint или внешний contract, меняющий acceptance scope. Implementation-предпочтения — в `plan`, не в `spec`.
- Spec должна быть достаточно содержательной, чтобы агент и reviewer поняли сценарий и границы scope без planning-артефактов.
- Не выполняйте git commit/push/tag, не открывайте PR без явной просьбы.
- Используйте `.speckeep/templates/spec.md` при создании. Настроенный язык документации. Языки внутри одной spec не смешивайте.

## Acceptance

Маркеры `Given`/`When`/`Then` канонические в любом языке: предусловие / действие / наблюдаемый результат. AC наблюдаемы и проверяемы, observable proof signal обязателен. Набор сильных критериев > длинный избыточный список. Короткая строка «почему важен» — когда это помогает planning.

## Правила качества секций

- `## Цель` — кто получает пользу, что меняется, что значит успех.
- `--greenfield`: `## User Stories` добавляйте только когда это реально снижает неоднозначность; держите 1–3 приоритезированные истории (`P1/P2/P3`), а не backlog целиком.
- `--greenfield`: `## MVP Slice` должен зафиксировать минимальный независимо полезный срез и какие `AC-*` он обязан закрыть первым.
- `--greenfield`: `## First Deployable Outcome` должен явно говорить, что можно показать или проверить руками после первого implementation pass.
- `## Why Now` — только когда момент/боль/бизнес-давление реально влияют на приоритет.
- `## Основной сценарий` — 3–5 конкретных шагов, не общих формулировок.
- `## Change Delta` — что становится новым, что меняется, что остаётся.
- `## Affected Surfaces` — только user/repo-visible surfaces, задающие границу.
- `## Scope Snapshot`/`## Scope`/`## Вне scope` — границы очевидны reviewer.
- `## Контекст` — репо-ограничения, сохраняемое поведение, точки интеграции, реально влияющие.
- `## Допущения` — каждое неочевидное допущение; пропуск — дефект.
- `## Критерии успеха` — `SC-*` с числом и методом измерения (напр., `SC-001 p95 < 1с при 100 конкурентных`).
- `## Требования` — чёткие и проверяемые (`RQ-001 CSV с UTF-8 BOM; строки совпадают с текущим фильтром`), без «нормально»/«адекватно».
- `## Краевые случаи` — только то, что меняет реализацию/проверку.
- `## Открытые вопросы` — `none`, если ничего не осталось.
- Не сливайте фичи, не прячьте scope в edge cases, не оставляйте AC как `TBD`, не добавляйте списки библиотек/версий если это не product/repo constraint.
- Плотность > длина; filler — дефект.

## Output

- Запишите/patch `.speckeep/specs/<slug>/spec.md` после подтверждения feature-ветки.
- Суммируйте цель, scope, AC, открытые вопросы.
- Завершайте summary block: `Slug`, `Status`, `Artifacts`, `Blockers`, `Готово к`.
- Готово: `Готово к: /speckeep.inspect <slug>`.

## Self-Check

- На feature-ветке, в рамках одной фичи, все AC со стабильным ID + Given/When/Then?
- Reviewer поймёт сценарий и границы без дополнительных объяснений?
