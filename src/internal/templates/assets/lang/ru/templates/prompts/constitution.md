# Prompt конституции SpecKeep

Вы создаете или обновляете конституцию проекта (путь задается `project.constitution_file`; по умолчанию: `.speckeep/constitution.md`).

## Goal

Сформируйте строгую конституцию проекта, которая является авторитетной и для людей, и для агентной разработки.

## Примечание о путях

Пути в этом промпте показаны для layout по умолчанию. Если в `.speckeep/speckeep.yaml` переопределены `project.constitution_file`, `paths.specs_dir` или `paths.archive_dir`, всегда следуйте путям из конфигурации, а не примерам по умолчанию.
Читайте `.speckeep/speckeep.yaml` максимум один раз за сессию для резолва путей; не перечитывайте его без необходимости (только если конфиг изменился или путь неоднозначен).

Для **greenfield-проекта** (с нуля) сфокусируйтесь на установлении незыблемых архитектурных границ, выборе технологического стека и определении стандартов качества, которые станут основой для последующего проектирования «фундамента» (Foundation) проекта.

Для существующей кодовой базы сначала формализуйте уже наблюдаемую реальность проекта, затем отдельно закрепите новые обязательные правила, явно заданные пользователем.

## Greenfield mode

Если проект создается с нуля:

- зафиксируйте выбранный технологический стек (язык, фреймворки, БД)
- определите ключевые архитектурные паттерны (напр. Clean Architecture, Hexagonal)
- установите стандарты именования и структуру каталогов
- определите правила работы с зависимостями и внешними интеграциями
- заполните секции `## Технологический стек` и `## Основная архитектура` с достаточной детализацией, чтобы они стали основой для последующих фаз spec и plan

## Brownfield mode

Если проект уже существует, работайте в двух слоях:

- `Observed Reality`
  - фиксируйте только то, что подтверждается структурой репозитория, конфигурацией, ключевыми entrypoint-файлами, зависимостями и существующей документацией
- `Declared Law`
  - фиксируйте новые обязательные правила разработки только тогда, когда они явно заданы пользователем или уже надежно подтверждаются проектом

Не перепроектируйте существующий проект под идеальную архитектуру. Сначала описывайте реальное состояние, затем формализуйте целевое управление изменениями.

## Load First

- текущий запрос пользователя и контекст разговора
- `.speckeep/constitution.md`
- верхнеуровневую структуру каталогов
- только минимально необходимый контекст репозитория, чтобы сделать конституцию конкретной

## Load If Present

- `README.md`
- `AGENTS.md`
- манифесты и конфигурации проекта, которые быстро объясняют язык, runtime, архитектурные границы или интеграции

## Repository Evidence

**Сильные сигналы** (безопасно выводить правила): границы каталогов (`api/`, `cmd/`, `internal/`, `migrations/`), зависимости и конфиги, раскрывающие transports/storage/runtime, существующие docs по workflow, entrypoint-файлы с composition root или разделением ролей.

**Слабые сигналы** (только контекст, не источник правил): единичные файлы без поддержки структурой, naming без подтверждения конфигами, общие best-practice ожидания без привязки к репозиторию.

## Do Not Read By Default

- большие участки кода, не влияющие на конституцию
- старые feature artifacts, если они не нужны для разрешения конституционного конфликта
- весь репозиторий целиком

## Stop Conditions

Остановитесь и задайте минимальный уточняющий вопрос, если:

- назначение проекта нельзя сформулировать конкретно
- правила процесса разработки пришлось бы угадывать, а не выводить из контекста
- виден конституционный конфликт, который нельзя разрешить по доступному контексту
- без уточнения пришлось бы объявить архитектурную границу, ownership или workflow как обязательные без достаточного evidence

Не расширяйте чтение репозитория без одной из этих причин.

Если конституция актуальна и не конфликтует с запросом — сообщите об этом и не изменяйте файл.

## Required behavior

- Работайте patch-обновлением существующего файла `.speckeep/constitution.md`.
- Сохраните эти обязательные секции без изменения их роли:
  - `## Назначение`
  - `## Ключевые принципы`
  - `## Ограничения`
  - `## Технологический стек`
  - `## Основная архитектура`
  - `## Приоритеты решений`
  - `## Ключевые измерения качества`
  - `## Языковая политика`
  - `## Процесс разработки`
  - `## Управление`
  - `## Протокол исключений`
  - `## Последнее обновление`
- Обеспечьте минимум 5 подразделов принципов под `## Ключевые принципы` с заголовками формата `### Название принципа`.
- Дополнительные секции можно добавлять, если они реально улучшают управление проектом.
- Заменяйте placeholder-токены вроде `[PROJECT_NAME]`, `[TECH_STACK]` или `[ARCHITECTURE]` конкретным содержимым.
- При использовании флага `--foundation` конституция становится авторитетной записью о том, на чём и как построен проект. Глубоко заполните секции `## Технологический стек` и `## Основная архитектура` так, чтобы они были actionable для всех downstream фаз. Эти секции должны описывать:
  - Выбранные языки, фреймворки, базы данных и инфраструктуру — с версионными ограничениями там, где они существенны.
  - Структурные паттерны, потоки данных, границы модулей и организацию каталогов.
  - Ограничения, которые являются non-negotiable для всех будущих spec, plan и implementation-решений.
- Для brownfield-проекта сначала закрепите то, что подтверждается кодовой базой, и только затем добавляйте новые обязательные правила.
- Если пользователь явно требует новые нормы разработки, закрепляйте их в `## Процесс разработки` и `## Управление` как обязательные правила следующей работы над проектом.
- Секция `## Процесс разработки` ДОЛЖНА определить, как feature-ветки, спецификации, inspect, планы, задачи и реализация соотносятся с соблюдением конституции.
- Секция `## Процесс разработки` ДОЛЖНА явно указать, при каких условиях spec, inspect, plan, tasks или implement нарушают конституцию и не могут быть продолжены.
- Секция `## Приоритеты решений` ДОЛЖНА коротко фиксировать 3-5 rule-like приоритетов для разрешения trade-off'ов, например простота против расширяемости, correctness против скорости доставки, поддерживаемость против cleverness.
- Секция `## Ключевые измерения качества` ДОЛЖНА фиксировать только project-relevant quality dimensions. Не делайте generic quality essay; ограничьтесь 3-5 короткими, проверяемыми bullets.
- Секция `## Протокол исключений` ДОЛЖНА объяснять, как проект фиксирует допустимые отклонения от конституции и когда downstream phases должны считать конфликт blocker.
- Не объявляйте DDD-boundary, event contract ownership, release policy или branch strategy обязательными, если это не подтверждено репозиторием или не задано пользователем явно.
- Если критической информации не хватает, задавайте только минимально необходимый уточняющий вопрос.
- Используйте строгий и проверяемый язык. Избегайте расплывчатых формулировок. Каждый принцип должен позволять однозначно ответить на вопрос «это решение конституции соответствует?».
- Не превращайте `## Приоритеты решений`, `## Ключевые измерения качества` или `## Протокол исключений` в длинний handbook. Предпочитайте компактные bullets, пригодные для downstream phase checks.
- Конституция — документ высшего приоритета. Спецификации, inspect-отчеты, планы, задачи и реализация обязаны ей соответствовать.

## Update rules

- Сохраняйте уже хорошие принципы, если они не конфликтуют с новыми требованиями.
- Предпочитайте patch-уточнение полному переписыванию. При уточнении принципа сохраняйте его проверяемость — не заменяйте конкретные правила расплывчатыми обобщениями.
- Если правило выведено из репозитория, формулируйте его как наблюдаемую устойчивую норму проекта, а не как абстрактную best practice.
- Если правило задано пользователем как новое обязательство, формулируйте его как law для последующих фаз.
- Обновляйте `## Последнее обновление` сегодняшней датой в формате `YYYY-MM-DD`, когда конституция меняется.

## Summary artifact

После записи или обновления `constitution.md` также запишите или обновите `.speckeep/constitution.summary.md`.

Summary ДОЛЖЕН содержать только:

- `## Purpose` — 1-2 предложения
- `## Key Constraints` — 3-5 bullets, только жёсткие неотменяемые ограничения
- `## Language Policy` — 3 строки: docs, agent, comments
- `## Development Workflow` — 3-5 ключевых правил, наиболее важных для фаз spec, plan и implement
- `## Decision Priorities` — 3-5 bullets

Держите summary не длиннее 60 строк. Он загружается в `implement`, `tasks` и `verify` вместо полной конституции, чтобы сократить context overhead. Summary не заменяет полную конституцию на фазах, где требуется constitutional consistency check (spec, inspect, plan).

## Проверка влияния обновления

После записи или patch-обновления `constitution.md` проверьте, не конфликтуют ли существующие активные спеки с изменёнными правилами:

- Запустите `.speckeep/scripts/list-specs.*` для перечисления активных спеков. Если скрипт недоступен, перечислите файлы в `.speckeep/specs/` напрямую для определения активных slugs.
- Для каждого активного спека сравните его `## Goal`, `## Scope` и `## Acceptance Criteria` с изменёнными секциями конституции.
- Если спек может конфликтовать с обновлённой конституцией, пометьте его: `NEEDS RE-INSPECT: <slug> — <причина>` в выводе.
- Не модифицируйте сами спеки — только сообщите, какие из них нуждаются в повторной проверке.
- Если активных спеков нет или ни один не конфликтует, укажите: «Обновление не затрагивает активные спеки.»

## Output expectations

- Запишите обновлённый `.speckeep/constitution.md` и `.speckeep/constitution.summary.md`
- Кратко опишите что изменилось и что осталось неразрешённым
- Укажите что было выведено из кодовой базы, а что добавлено как новое обязательное правило
- Неразрешённые конституционные вопросы помечайте как **BLOCKER** для downstream фаз
- Когда можно продолжать: `Готово к: /speckeep.spec`
