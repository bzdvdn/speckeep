# Prompt конституции SpecKeep

Создаёте или обновляете конституцию проекта (путь — `project.constitution_file`; по умолчанию: `.speckeep/constitution.md`).

Следуйте базовым правилам в `AGENTS.md` (пути, git, load discipline, readiness scripts, язык, phase discipline).

## Goal

Строгая конституция проекта, авторитетная и для людей, и для агентов.

Для **greenfield-проекта** — незыблемые архитектурные границы, технологический стек, стандарты качества как основа для последующего Foundation-дизайна.

Для существующего репозитория — сначала формализуйте observable reality, потом отдельно закрепите новые обязательные правила, явно заданные пользователем.

## Greenfield mode

- зафиксируйте технологический стек (язык, фреймворки, БД)
- определите архитектурные паттерны (Clean Architecture, Hexagonal и т.п.)
- задайте соглашения по именованию и структуру каталогов
- определите правила зависимостей и внешних интеграций
- заполните `## Технологический стек` и `## Основная архитектура` с достаточной детализацией как опору для spec/plan

## Brownfield mode

Два слоя:

- `Observed Reality` — только то, что подтверждается структурой репо, конфигурацией, entrypoints, зависимостями, существующими docs
- `Declared Law` — новые обязательные правила только когда пользователь явно просит или они уже надёжно подтверждены проектом

Не перепроектируйте проект под идеальную архитектуру. Сначала текущее состояние, потом формализация управления изменениями.

## Load First

- текущий запрос пользователя и контекст разговора
- `.speckeep/constitution.md`
- верхнеуровневая структура каталогов
- минимальный repo-контекст, нужный для конкретики

## Load If Present

- `README.md`, `AGENTS.md`
- манифесты/конфиги, объясняющие язык, runtime, архитектурные границы или интеграции

## Repository Evidence

**Сильные сигналы** (безопасно выводить правила): границы каталогов (`api/`, `cmd/`, `internal/`, `migrations/`), зависимости/конфиги, показывающие transports/storage/runtime, существующие workflow docs, entrypoints с composition root или разделением ролей.

**Слабые сигналы** (только контекст, не источник правил): единичные файлы без поддержки структурой, naming без подтверждения конфигами, generic best practice без привязки к репо.

## Do Not Read By Default

- большие области кода, не влияющие на конституцию
- старые feature artifacts — кроме разрешения конституционного конфликта
- репозиторий целиком

## Stop Conditions

Один минимальный уточняющий вопрос только если:

- назначение проекта нельзя сформулировать конкретно
- правила процесса пришлось бы угадывать, а не выводить
- виден конституционный конфликт, который нельзя разрешить
- без уточнения пришлось бы объявить архитектурные границы/ownership/workflow обязательными без evidence

Не расширяйте чтение репо без одной из этих причин.

Конституция уже актуальна и не конфликтует с запросом — сообщите и не меняйте файл.

## Required behavior

- Patch-обновление существующего `.speckeep/constitution.md`.
- Сохраните обязательные секции:
  - `## Назначение`, `## Ключевые принципы`, `## Ограничения`, `## Технологический стек`, `## Основная архитектура`, `## Приоритеты решений`, `## Ключевые измерения качества`, `## Языковая политика`, `## Процесс разработки`, `## Управление`, `## Протокол исключений`, `## Последнее обновление`
- `## Ключевые принципы` — ≥ 5 подразделов с заголовками `### Название принципа`.
- Дополнительные секции — если реально улучшают управление.
- Заменяйте placeholder-токены (`[PROJECT_NAME]`, `[TECH_STACK]`, `[ARCHITECTURE]`) конкретным содержимым.
- Флаг `--foundation`: конституция — авторитетная запись о том, на чём и как построен проект. Глубоко заполните `## Технологический стек` и `## Основная архитектура`:
  - языки, фреймворки, БД, инфраструктура с version constraints где важно
  - структурные паттерны, data flows, границы модулей, организация каталогов
  - non-negotiable constraints для всех будущих spec/plan/implementation
- Brownfield: сначала то, что уже подтверждено кодовой базой, потом новые нормы.
- Новые правила от пользователя → в `## Процесс разработки` и `## Управление` как обязательные.
- `## Процесс разработки` ДОЛЖНА определить связь feature-веток, спец, inspect, планов, задач и реализации с соблюдением конституции.
- `## Процесс разработки` ДОЛЖНА явно указать условия, при которых spec/inspect/plan/tasks/implement нарушают конституцию и не могут быть продолжены.
- `## Приоритеты решений` — 3-5 rule-like приоритетов для trade-off'ов (простота vs расширяемость, correctness vs скорость, maintainability vs cleverness).
- `## Ключевые измерения качества` — 3-5 коротких проверяемых bullets, только project-relevant, не generic-эссе.
- `## Протокол исключений` — как фиксируются допустимые отклонения и когда downstream-phase считают конфликт blocker.
- Не объявляйте DDD-boundary, event-contract ownership, release policy, branch strategy обязательными без подтверждения репо или явной просьбы.
- Строгий, проверяемый язык. Каждый принцип — «это решение конституции соответствует?» с однозначным ответом.
- `## Приоритеты решений`/`## Ключевые измерения качества`/`## Протокол исключений` не превращайте в handbook — компактные bullets, пригодные для downstream checks.
- Конституция — документ высшего приоритета; спеки, inspect, планы, задачи, реализация обязаны ей соответствовать.

## Update rules

- Сохраняйте уже хорошие принципы, если они не конфликтуют.
- Предпочитайте patch полному переписыванию. Уточняя принцип, сохраняйте проверяемость — не заменяйте конкретные правила расплывчатыми обобщениями.
- Правило выведено из репо → формулируйте как наблюдаемую норму. Правило от пользователя → формулируйте как law.
- Обновляйте `## Последнее обновление` текущей датой в формате `YYYY-MM-DD`.

## Summary artifact

После записи/patch `constitution.md` запишите/обновите `.speckeep/constitution.summary.md` только с:

- `## Purpose` — 1-2 предложения
- `## Key Constraints` — 3-5 bullets, только жёсткие неотменяемые ограничения
- `## Language Policy` — 3 строки: docs, agent, comments
- `## Development Workflow` — 3-5 ключевых правил для spec/plan/implement
- `## Decision Priorities` — 3-5 bullets

≤ 60 строк. Загружается `implement`/`tasks`/`verify` вместо полной конституции. Не заменяет полную конституцию на фазах с constitutional consistency check (spec, inspect, plan).

## Проверка влияния обновления

После записи/patch проверьте конфликты с активными спеками:

- Запустите `.speckeep/scripts/list-specs.*`; fallback — перечислите `.speckeep/specs/`.
- Для каждого активного спека сравните `## Goal`, `## Scope`, `## Acceptance Criteria` с изменёнными секциями.
- Конфликт → `NEEDS RE-INSPECT: <slug> — <причина>`. Не модифицируйте спеки.
- Конфликтов нет → «Обновление не затрагивает активные спеки.»

## Output

- Запишите обновлённый `.speckeep/constitution.md` и `.speckeep/constitution.summary.md`.
- Кратко — что изменилось и что осталось неразрешённым; что выведено из репо vs добавлено как новое обязательное правило.
- Неразрешённые вопросы → **BLOCKER** для downstream.
- Готово: `Готово к: /speckeep.spec`.
