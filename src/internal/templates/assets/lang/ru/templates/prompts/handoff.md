# Prompt передачи сессии SpecKeep

Вы создаёте документ передачи сессии для одной фичи.

## Goal

Сформируйте компактный документ передачи в `.speckeep/handoff/<slug>.md`, позволяющий новой агентской сессии возобновить работу над фичей без потери контекста.

## Phase Contract

Inputs: текущие артефакты фичи `<slug>` (подходящее для текущей фазы подмножество).
Outputs: `.speckeep/handoff/<slug>.md` с текущей фазой, завершёнными работами, открытыми задачами, ключевыми решениями и следующей командой.
Stop if: slug неоднозначен или для slug не существует артефактов фичи.

## Load First

Всегда сначала читайте эти файлы, если они существуют:

- `.speckeep/constitution.md`
- `.speckeep/specs/<slug>/spec.md`

## Load If Present

Читайте в зависимости от определённой фазы — только артефакты, которые существуют и вносят вклад в handoff:

- `.speckeep/specs/<slug>/inspect.md` — читать строку verdict для заполнения `## Current Phase` и обнаружения blockers
- `.speckeep/specs/<slug>/plan/plan.md` — читать `DEC-*` записи для заполнения `## Key Decisions`, когда фича в фазе plan или позже
- `.speckeep/specs/<slug>/plan/tasks.md` — читать чекбоксы задач для заполнения `## Open Work` и `## Completed`, когда фича в фазе tasks/implement или позже
- `.speckeep/specs/<slug>/plan/verify.md` — читать metadata block и строку verdict для заполнения `## Current Phase` и обнаружения verification concerns или blockers

## Do Not Read By Default

- `.speckeep/specs/<slug>/plan/data-model.md`
- `.speckeep/specs/<slug>/plan/contracts/`
- `.speckeep/specs/<slug>/plan/research.md`
- нерелевантные specs или plan packages
- implementation-файлы
- исходники скриптов

## Stop Conditions

Остановитесь и задайте вопрос только если:

- slug неоднозначен и не может быть определён из контекста или аргументов
- для slug не существует никаких артефактов фичи
- вызов без slug, и активных фич вообще нет

## Режим всех фич

При вызове без slug (в аргументах пользователя нет slug):
- Запустите `.speckeep/scripts/list-specs.*` для перечисления активных фич; исходник не читайте.
- Для каждой активной фичи сформируйте документ передачи в `.speckeep/handoff/<slug>.md`, используя те же правила, что и для одной фичи.
- Выведите краткую inline-сводку: одна строка на фичу с slug, фазой и следующей командой.
- Всегда перезаписывайте существующие handoff-файлы — каждый является снимком текущего состояния.

## Rules

- Определяйте текущую фазу по набору присутствующих артефактов: нет spec → до spec; spec есть, нет inspect → spec; inspect есть, нет plan → inspect; plan есть, нет tasks → plan; tasks есть с открытыми пунктами → tasks или implement; все tasks закрыты, нет `verify.md` → verify; `verify.md` существует с вердиктом `pass` или `concerns` → archive; `verify.md` существует с вердиктом `blocked` → возврат к фазе, указанной в строке `Return to:` отчёта (не предлагайте archive).
- Если доступен `.speckeep/scripts/list-open-tasks.*`, запускайте его для перечисления незавершённых задач; опирайтесь на его вывод, а не читайте исходник скрипта.
- Держите каждый документ передачи компактным: он должен загружаться за один дешёвый read в начале новой сессии. Учтите, что генерация handoff в режиме всех фич по своей природе дороже, чем для одного slug — это ожидаемо и допустимо; принцип «cheap read» применяется к каждому созданному документу, а не к процессу их генерации.
- Не воспроизводите полное содержимое артефактов в документе передачи — указывайте пути к файлам, а не их содержимое.
- Ссылайтесь на открытые пункты по стабильным ID (T*, AC-*, DEC-*, RQ-*) там, где они доступны.
- Каждая секция должна нести полезный сигнал; опускайте секции, которые были бы пустыми или избыточными.
- При записи в файл используйте настроенный язык документации проекта.
- Добавляйте machine-readable metadata block в начало файла.

## Output Structure

Записывайте `.speckeep/handoff/<slug>.md` со следующей структурой:

- YAML metadata block: `report_type`, `slug`, `phase`, `docs_language`, `generated_at`
- `# Handoff: <slug>`
- `## Current Phase` — текущая фаза фичи и основание для такого вывода
- `## Completed` — присутствующие артефакты и закрытые задачи (по ID, если доступны)
- `## Open Work` — оставшиеся задачи или отсутствующие обязательные артефакты, со стабильными ID там, где возможно
- `## Key Decisions` — решения из `plan.md`, существенно влияющие на оставшуюся работу (ID DEC-*)
- `## Open Questions` — блокеры или нерешённые вопросы, которые должны быть закрыты перед следующей фазой
- `## Artifacts` — точные project-relative пути ко всем релевантным файлам этого slug
- `## Next Command` — точная slash-команда и slug для немедленного возобновления работы

## Output Expectations

- Всегда перезаписывайте `.speckeep/handoff/<slug>.md`, если файл уже существует — handoff является снимком текущего состояния, и предыдущая версия мгновенно устаревает.
- Записывайте файл в `.speckeep/handoff/<slug>.md`.
- Также выводите краткое inline-резюме в разговоре: текущая фаза, количество открытых пунктов и готово к.
- Завершайте разговор точной строкой `Готово к`, чтобы новая сессия могла начать без повторного чтения.
- Если открытые вопросы блокируют следующую фазу, указывайте это в `## Next Command` вместо имени phase command.

## Self-Check

- Правильно ли определена текущая фаза по набору присутствующих артефактов?
- Ссылаются ли открытые пункты на стабильные ID там, где это возможно?
- Достаточно ли компактен документ для дешёвой загрузки в начале новой сессии?
- Содержит ли `## Next Command` точную slash-команду и slug для возобновления?
- Удалось ли избежать воспроизведения полного содержимого артефактов в документе передачи?
