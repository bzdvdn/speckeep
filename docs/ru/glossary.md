# Glossary

## Constitution

Главный документ проекта. Он задает неоспоримые правила для архитектуры, workflow, языковой политики и governance. Для новых проектов также служит техническим фундаментом, фиксируя технологический стек и базовую архитектуру.

## Spec

Документ уровня фичи, который описывает, что должно быть построено и зачем.

## Inspect

Фаза проверки, которая смотрит spec и связанные артефакты на полноту, согласованность и соответствие конституции.

## Plan

Техническая фаза проектирования, которая переводит одну spec в артефакты, пригодные для реализации.

## Plan Package

Набор файлов под `.speckeep/specs/<slug>/plan/`, обычно включающий `plan.md`, `tasks.md`, `data-model.md`, `contracts/` и optional `research.md`.

## Tasks

Исполнимая декомпозиция feature plan. В SpecKeep файл `tasks.md` живет внутри plan package.

## Implement

Фаза исполнения, которая стартует от незавершенных задач и обновляет состояние tasks.

## Verify

Легкая post-implementation проверка, которая подтверждает, что фича достаточно согласована с tasks, spec, plan artifacts и правилами проекта, чтобы безопасно двигаться к archive или заявлению о завершенности.

## Archive

Фаза и область хранения, которые используются для сохранения исторического снимка завершенного или неактивного feature package.

## Data Model

Plan-артефакт, который описывает сущности, структуры, связи и важные invariants, нужные для реализации.

## Contracts

Plan-артефакты, определяющие интерфейсы, например API, события или другие внешние границы взаимодействия.

## Research

Опциональный plan-артефакт, который нужен только тогда, когда нужно сохранить неопределенность, внешнее исследование или архитектурный tradeoff.

## Agent Target

Поддерживаемая агентная экосистема, для которой SpecKeep может генерировать project-local command или prompt files, например `claude`, `codex`, `copilot`, `cursor`, `kilocode` или `trae`.

## Orphaned Agent Artifact

Сгенерированный agent file, который все еще лежит на диске, хотя его target уже не включен в `.speckeep/speckeep.yaml`.

## Docs Language

Настроенный язык, который используется для генерируемой проектной документации, такой как `constitution.md`, specs, plans и tasks.

## Agent Language

Настроенный язык, который используется для generated prompts и guidance в `AGENTS.md`.

## Comments Language

Настроенный предпочитаемый язык для новых или измененных комментариев в коде во время реализации.

## Given / When / Then

Канонические BDD-маркеры, которые используются в acceptance criteria независимо от языка документации. Текст вокруг них при этом может следовать настроенному docs language.

## Acceptance ID

Стабильный идентификатор критерия приемки, например `AC-001`. Он помогает делать traceability явным между spec, tasks и inspect-отчетами.

## Traceability (Прослеживаемость)

Практика связывания кода реализации и тестов с задачами и критериями приемки с помощью стабильных ID. В SpecKeep это реализуется через аннотации `@sk-task` и `@sk-test`.

## Lazy Decomposition (Ленивая декомпозиция)

Стратегия минимизации раздувания контекста: задачи на старте создаются на высоком уровне, а агент реализации выполняет **In-place Декомпозицию** только для текущей активной задачи.

## In-place Декомпозиция

Процесс уточнения сложной задачи путем добавления вложенных подзадач (например, `T1.1.1`) прямо в `tasks.md` на фазе реализации.

## Smart Branching (Умное ветвление)

Функция, обеспечивающая согласованность разработки в Git путем проверки соответствия текущей ветки ожидаемому шаблону `feature/<slug>`.
